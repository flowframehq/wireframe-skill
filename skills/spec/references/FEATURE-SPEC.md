# 예제: 기능 명세 — AUTH 도메인

2단계 깊이(LOGIN > EMAIL_LOGIN, SOCIAL_LOGIN)를 가진 도메인 파일의 완성 예제.

---

## 파일: `docs/features/AUTH.md`

```markdown
---
domain: AUTH
label: 인증
toc:
  - id: LOGIN
    label: 로그인
    children:
      - id: EMAIL_LOGIN
        label: 이메일 로그인
      - id: SOCIAL_LOGIN
        label: 소셜 로그인
  - id: SIGNUP
    label: 회원가입
---

# 인증

## 로그인

### 와이어프레임 요소

| id | 요소 | type | 설명 |
|----|------|------|------|
| EMAIL | 이메일 | input | 이메일 주소 입력 필드 |
| PASSWORD | 비밀번호 | input | 비밀번호 입력 필드 (마스킹) |
| SUBMIT | 로그인 버튼 | button | 인증 요청 주요 액션 버튼 |

### 상태

| 상태 | 설명 |
|------|------|
| 기본 | 폼 비어있음, 로그인 버튼 활성 |
| 에러 | "이메일 또는 비밀번호가 올바르지 않습니다" 메시지 |
| 로딩 | 로그인 버튼 비활성 + 스피너 |

### 인터랙션

- 로그인 버튼 클릭 → 인증 API 호출 → 성공 시 대시보드로 이동
- 이메일 형식 오류 시 즉시 인라인 에러 표시

### 비즈니스 로직

- 이메일 형식 검증
- 5회 실패 시 30분 잠금
- 비밀번호 최소 8자, 영문+숫자

### 이메일 로그인

#### 와이어프레임 요소

| id | 요소 | type | 설명 |
|----|------|------|------|
| EMAIL | 이메일 입력 | input | 이메일 주소 입력 필드 |
| CODE | 인증 코드 입력 | input | 이메일로 전송된 6자리 인증 코드 입력 |
| REQUEST | 인증 요청 버튼 | button | 이메일 인증 코드 발송 요청 |
| VERIFY | 확인 버튼 | button | 인증 코드 검증 및 로그인 완료 |

#### 상태

| 상태 | 설명 |
|------|------|
| 이메일 입력 | 이메일 입력 대기, 인증 요청 버튼 활성 |
| 코드 입력 대기 | 이메일 전송 완료, 코드 입력 필드 활성 |
| 인증 실패 | "인증 코드가 올바르지 않습니다" 메시지 |

#### 인터랙션

- 인증 요청 버튼 클릭 → 이메일로 인증 코드 발송
- 확인 버튼 클릭 → 코드 검증 → 성공 시 로그인 완료

### 소셜 로그인

#### 와이어프레임 요소

| id | 요소 | type | 설명 |
|----|------|------|------|
| GOOGLE | 구글 로그인 | button | Google OAuth 로그인 버튼 |
| KAKAO | 카카오 로그인 | button | Kakao OAuth 로그인 버튼 |
| NAVER | 네이버 로그인 | button | Naver OAuth 로그인 버튼 |

#### 인터랙션

- 소셜 버튼 클릭 → OAuth 제공자 팝업 → 인증 완료 시 대시보드 이동

## 회원가입

### 와이어프레임 요소

| id | 요소 | type | 설명 |
|----|------|------|------|
| EMAIL | 이메일 | input | 가입용 이메일 주소 |
| PASSWORD | 비밀번호 | input | 비밀번호 입력 (마스킹) |
| PASSWORD_CONFIRM | 비밀번호 확인 | input | 비밀번호 재입력 확인 |
| TERMS_AGREE | 이용약관 동의 | checkbox | 필수 약관 동의 체크박스 |
| SUBMIT | 가입 버튼 | button | 회원가입 실행 |

### 상태

| 상태 | 설명 |
|------|------|
| 기본 | 폼 비어있음 |
| 검증 에러 | 비밀번호 불일치 또는 이메일 중복 메시지 |
| 가입 완료 | 환영 메시지 + 로그인 화면 이동 안내 |

### 인터랙션

- 가입 버튼 클릭 → 검증 → 성공 시 환영 화면
- 이메일 입력 후 포커스 이탈 → 중복 체크

### 비즈니스 로직

- 이메일 중복 불가
- 비밀번호 최소 8자, 영문+숫자+특수문자
- 14세 미만 가입 불가
```

## 핵심 패턴

| 패턴 | 이 예제에서 |
|------|-----------|
| 도메인 = 플랫 파일 | `AUTH.md` 하나에 LOGIN, SIGNUP 등 모든 기능 포함 |
| TOC가 단일 소스 | frontmatter `toc`의 구조 = 본문 헤딩 구조 |
| 2단계 깊이 | LOGIN(H2) > EMAIL_LOGIN(H3), SOCIAL_LOGIN(H3) |
| 예약 섹션 vs 하위 기능 | `와이어프레임 요소`(예약)은 콘텐츠, `이메일 로그인`(비예약)은 하위 기능 |
| 콘텐츠 섹션 배치 | 기능 헤딩 바로 아래 뎁스에 `와이어프레임 요소`, `상태` 등 배치 |
| 최소 기준 | `와이어프레임 요소` 테이블만 있어도 유효 |
| 요소 ID 원천 | `와이어프레임 요소` 테이블의 `id` 컬럼을 wireframer가 그대로 사용 |
| 요소 type | `input`, `button`, `checkbox`, `link` 등 표준 타입 사용 |
| featureId 파생 | AUTH + LOGIN + EMAIL_LOGIN → `AUTH__LOGIN__EMAIL_LOGIN` |
| INDEX.md 동기화 | 도메인 파일 생성 시 INDEX.md에 등록 |
