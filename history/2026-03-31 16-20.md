## 목표 (Goal)

## 핵심 결정 사항 (Key Decisions)

- **2-tier DOM tracking**: `data-feature-id`(코멘트 앵커) + `data-feature`(하이라이트) 분리. FlowFrame에서 피처 클릭 시 코멘트, 요소 호버 시 하이라이트.
- **`data-feature-label`**: 피처 래퍼에 한글 표시명 포함. FlowFrame UI에서 label을 바로 읽을 수 있도록.
- **tree 메타데이터 (v1.1)**: `elements[]` flat → `features[].elements[]` 트리. featureId/label/spec 중복 제거. 파일 크기 56KB→52KB.
- **review-tab `:has()` CSS**: `peer-checked`는 label이 nested여서 안 먹음. `:has(> input:nth-of-type(N):checked)` 위치 기반 매칭으로 교체. 패널 전환은 `peer-checked:block` 유지 (형제 관계라 동작).
- **재귀 트리 설계 합의**: branch는 `children`, leaf는 `elements`. 둘 중 하나만 가짐. `data-feature-id` 중첩 허용 (branch→leaf). 5뎁스까지 표현 가능.

### 재귀 트리 설계 상세

```
메타데이터 구조:
- branch node: { featureId, label, spec(branch index.md), children: [Feature...] }
- leaf node:   { featureId, label, spec(leaf index.md), elements: [Element...] }
- children과 elements는 상호 배타 (oneOf)

HTML DOM 구조:
- branch: <div data-feature-id="MEMBER_ACTIONS" data-feature-label="회원 관리 액션">
            <div data-feature-id="MEMBER_ACTIONS__EDIT_INFO" data-feature-label="정보 수정">
              <div data-feature="FEATURE_MEMBER_ACTIONS__EDIT_INFO_TRIGGER">...</div>
            </div>
          </div>
- leaf at root: <div data-feature-id="MEMBER_PROFILE" data-feature-label="회원 기본 정보">
                  <div data-feature="FEATURE_MEMBER_PROFILE_ID">...</div>
                </div>

5뎁스 예시:
AUTH → AUTH__2FA → AUTH__2FA__METHODS → AUTH__2FA__METHODS__TOTP → AUTH__2FA__METHODS__TOTP__RECOVERY
```

## 주의사항 (Caveats)

### 재귀 트리 도입 시 걱정 포인트 (사용자에게 공유 필요)

1. **FlowFrame 파서 호환성**: FlowFrame 플랫폼이 `children` 재귀 구조를 파싱할 수 있는지 확인 필요. 기존에 flat `features[]`만 기대했다면 플랫폼 코드도 수정해야 함.

2. **코멘트 스코핑 모호성**: branch에도 코멘트가 달리고, 그 안의 leaf에도 코멘트가 달리면 — "이 코멘트는 branch 전체에 대한 건가, 특정 leaf에 대한 건가?" 구분이 필요. FlowFrame UI에서 이 구분을 어떻게 보여줄지 결정해야 함.

3. **review-tab `:has()` 중첩 간섭**: branch 안에 leaf가 있고, 둘 다 `.feature-review`이면 `:has(> input:nth-of-type)` 규칙이 outer branch의 radio를 inner leaf의 label에 잘못 매칭할 수 있음. `>` (direct child) 셀렉터로 방어하고 있지만, branch가 review-tab을 가지면서 동시에 child leaf도 review-tab을 가지는 케이스를 테스트해야 함.

4. **와이어프레임 생성 복잡도 증가**: 스킬을 쓰는 AI 에이전트가 재귀 구조를 올바르게 중첩 생성하는 난이도가 올라감. flat 구조는 순서대로 나열하면 됐지만, 재귀는 branch 래퍼를 먼저 만들고 그 안에 leaf를 넣어야 함. 스킬 지시문이 이를 명확히 가이드해야 함.

5. **depth 1 leaf (branch 없는 독립 피처)의 취급**: `MEMBER_PROFILE`처럼 branch 없이 바로 leaf인 경우와, `MEMBER_ACTIONS__EDIT_INFO`처럼 branch 안의 leaf인 경우가 혼재. 일관성 있게 처리하는 규칙 필요.

### 기타

- 현재 `group/feature` 같은 fixed literal 대신 `group/{feature-key}` 사용 규칙이 있음 — 재귀 중첩에서도 각 feature-review의 group name이 고유해야 hover reveal이 독립적으로 동작
- `:has()` CSS는 모든 모던 브라우저에서 지원되지만, 매우 오래된 브라우저에서는 동작 안 함 (FlowFrame이 모던 브라우저만 지원하면 문제 없음)
- `version`을 `"1.1"` → `"1.2"` 등으로 올려서 재귀 구조 도입 시점을 명확히 표기할 것
