# 상태 변경

## 와이어프레임 요소

| 요소 | type | 설명 |
|------|------|------|
| 현재 상태 표시 | text | 현재 주문 상태 뱃지 (크게 표시) |
| 다음 상태 버튼 그룹 | button | 현재 상태에서 전이 가능한 다음 상태 버튼들 |
| 사유 입력 | input | 상태 변경 사유 텍스트 영역 (취소/환불 시 필수, 그 외 선택) |
| 확인 다이얼로그 | dialog | 되돌릴 수 없는 변경(취소, 환불완료) 시 최종 확인 |

## 상태

| 상태 | 설명 |
|------|------|
| 기본 | 현재 상태에 따라 가능한 다음 상태 버튼 표시 |
| 처리 중 | 버튼 비활성화 + 스피너 |
| 성공 | 상태 변경 완료 토스트 메시지, 드로어 내 정보 갱신 |
| 실패 | "상태 변경에 실패했습니다" 에러 메시지 + 재시도 |
| 최종 상태 | 배송완료/취소/환불완료 후 더 이상 전이 불가 시 버튼 없음 |

## 인터랙션

- 다음 상태 버튼 클릭 → 취소/환불이면 사유 입력 + 확인 다이얼로그, 그 외는 바로 처리
- 확인 다이얼로그에서 "확인" → 상태 변경 실행
- 확인 다이얼로그에서 "취소" → 변경 취소, 이전 상태 유지
- 상태 변경 성공 → 주문 목록 테이블의 해당 행 상태도 즉시 갱신

## 유저스토리

- 쇼핑몰 관리자로서 주문 상태를 다음 단계로 진행시키고 싶다, 주문 처리 워크플로를 이행하기 위해
- 쇼핑몰 관리자로서 잘못된 상태 전이를 방지하고 싶다, 운영 실수를 줄이기 위해

## 인수조건

- Given 결제완료 상태 주문을 열면 When 상태 변경 영역을 보면 Then [배송준비], [취소] 버튼이 표시된다
- Given 배송중 상태 주문을 열면 When 상태 변경 영역을 보면 Then [배송완료] 버튼만 표시된다
- Given [취소] 버튼을 클릭하면 When 사유 입력 없이 확인을 누르면 Then 사유 필수 안내가 표시된다
- Given 상태 변경이 성공하면 When 드로어와 목록을 확인하면 Then 양쪽 모두 변경된 상태가 반영되어 있다
- Given 배송완료 상태이고 환불요청이 아니면 When 상태 변경 영역을 보면 Then 전이 가능 버튼이 없다

## 비즈니스 로직

- 현재 상태에서 허용되는 전이만 버튼으로 노출 (공통 비즈니스 로직의 전이 규칙 따름)
- 취소/환불 상태 변경 시 사유 입력 필수
- 그 외 상태 변경 시 사유 입력 선택
- 되돌릴 수 없는 상태 변경 (취소, 환불완료) 시 확인 다이얼로그 필수
